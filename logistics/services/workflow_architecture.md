# Logistics Workflow Upgrade Architecture

## 1. Recommended Modules and Relationships

- Client Module (existing): master party data shared by logistics and trade.
- Shipment Workflow Module (new): `ShipmentWorkflow` parent aggregate.
- Cargo Module (new): `CargoItemWorkflow` child entities under shipment.
- Transit Module (existing + integrated): reads workflow statuses and can trigger transitions through service layer.
- Inventory Module (new ledger core):
  - `InventoryPosition` (single source of truth balances)
  - `InventoryMovement` (append-only movement journal)
- Billing Module (new operational billing):
  - `BillingCharge` (event-driven charge lines)
  - `BillingInvoice` / `BillingInvoiceLine` / `BillingPayment`
- Workflow/Audit/Event Module (new):
  - `DomainEvent` (outbox + idempotency)
  - `WorkflowTransitionLog` (immutable transition audit)

### Relationship Summary

- `Client 1..N ShipmentWorkflow`
- `ShipmentWorkflow 1..N CargoItemWorkflow`
- `CargoItemWorkflow 1..1 InventoryPosition`
- `CargoItemWorkflow 1..N InventoryMovement`
- `ShipmentWorkflow 1..N BillingCharge`
- `CargoItemWorkflow 0..N BillingCharge`
- `ShipmentWorkflow 1..N BillingInvoice`
- `BillingInvoice 1..N BillingInvoiceLine`
- `BillingInvoice 1..N BillingPayment`

## 2. Data Flow Between Modules

1. Cargo received

- Create shipment + cargo item.
- Signal initializes `InventoryPosition` and writes `InventoryMovement(RECEIVED)`.

2. Cargo allocated

- Call `transition_cargo_item(..., to_status="ALLOCATED")`.
- Inventory moves `WAREHOUSE -> RESERVED`.
- Handling charge generated.
- Transition log and domain event recorded.

3. Cargo loaded

- Call `transition_cargo_item(..., to_status="LOADED")`.
- Inventory moves `RESERVED -> IN_TRANSIT`.
- LCL freight charge generated by chargeable weight.
- For FCL, shipment-level loaded transition can generate flat charge.

4. In transit / arrived / delivered

- Controlled by state machine service.
- Partial delivery allowed for LCL using `delivered_quantity`.
- Inventory moves `IN_TRANSIT -> DELIVERED` by delivered amount.
- On completion, invoice is issued/refreshed from open charges.

5. Payment

- Register payment using billing service.
- Invoice balance and status auto-updated.

## 3. Where to Enforce Business Rules

- Models:
  - Core invariants and constraints (`CheckConstraint`, immutable identifiers, non-negative balances).
- Service layer (primary):
  - State transitions and side effects in one transaction.
  - Inventory and finance side effects.
  - Idempotency enforcement.
- Signals (safety only):
  - Initialization events like first inventory position for newly created cargo.
- Domain events:
  - Durable outbox entries and correlation IDs for async consumers.

## 4. Operational vs Financial Data Separation

- Operational data:
  - `ShipmentWorkflow`, `CargoItemWorkflow`, `InventoryPosition`, `InventoryMovement`, `DomainEvent`, `WorkflowTransitionLog`.
- Financial data:
  - `BillingCharge`, `BillingInvoice`, `BillingInvoiceLine`, `BillingPayment`.
- Link boundary:
  - Financial records reference operational entities by FK and trigger event keys.
  - Financial totals are derived from `BillingCharge` journal, not manual free-entry.

## 5. State Machine Definition

### Shipment (FCL/LCL aggregate)

- `RECEIVED -> VERIFIED -> ALLOCATED -> LOADED -> IN_TRANSIT -> ARRIVED -> DELIVERED`
- Cancel path: `RECEIVED|VERIFIED|ALLOCATED -> CANCELLED`

### Cargo Item (supports LCL partial delivery)

- `RECEIVED -> VERIFIED -> ALLOCATED -> LOADED -> IN_TRANSIT -> ARRIVED -> PARTIALLY_DELIVERED -> DELIVERED`
- Also: `ARRIVED -> DELIVERED`
- Cancel path: `RECEIVED|VERIFIED|ALLOCATED -> CANCELLED`

## 6. Partial Delivery (Groupage)

- Use `transition_cargo_item(..., to_status="PARTIALLY_DELIVERED", delivered_quantity=X)`.
- Service enforces:
  - `0 < X <= remaining_quantity`.
  - Inventory movement only for delivered portion.
  - Cargo remains `PARTIALLY_DELIVERED` until cumulative delivered equals total.

## 7. Event-Driven Consistency and Idempotency

- Every transition writes a unique `DomainEvent.idempotency_key`.
- Side effects (inventory movement, charge generation) use derivative idempotency keys.
- Replays are safe because duplicate keys no-op.

## 8. Transaction Safety

- Transition services wrap all changes in `transaction.atomic()`.
- `select_for_update()` locks shipment/cargo/inventory rows to prevent race conditions.
- No status update is committed unless inventory + billing side effects succeed.

## 9. Reconciliation and Repair

- Command: `python manage.py reconcile_workflow_consistency`
- Auto-fix safe issues: `python manage.py reconcile_workflow_consistency --fix`

Checks include:

- Inventory balance total vs cargo total.
- Delivered status vs delivered quantity.
- Invoice line sum vs invoice totals.

## 10. DRF Layer Guidance

Use thin API views that call service methods:

- `POST /api/workflow/shipments/{id}/transition`
- `POST /api/workflow/cargo/{id}/transition`
- `POST /api/workflow/invoices/{id}/payments`

Serializer validation should only do shape/type checks.
State and side effects must stay in service layer to avoid coupling and inconsistency.

## 11. Commission Ledger (Director / Admin)

The `Commission` model is an **executive-only side ledger**, intentionally decoupled from the operational and billing pipelines above. It is not part of the shipment state machine and does not generate inventory movements or billing charges.

- **Scope**: Records commissions GMI Terralink earns per `Client` engagement.
- **Access**: Restricted to `is_superuser`, role `ADMIN` or role `DIRECTOR` via the `director_required` decorator. All four CRUD views (`commission_list`, `commission_create`, `commission_update`, `commission_delete`) enforce this.
- **Aggregation**: `DirectorReportingService.commission_totals()` returns per-currency totals (`USD`, `UGX`, `CNY`, `EUR`, `GBP`, `KES`) for the executive reports dashboard.
- **Audit**: Every create / update / delete is written via `log_audit("commission", action, ...)`.
- **Boundaries**: Commission entries are free-entry by the Director / Admin (similar to other manual finance ledgers) and do **not** auto-derive from `BillingCharge` or shipment events. Linking commissions to specific shipments or POs would be a future enhancement and should flow through a service method, not direct ORM writes.
